# 公平回溯训练协议

更新时间：2026-04-29

本文定义项目中的回溯训练机制。它的定位不是让 AI 获得超出玩家的信息，也不是把回溯作为正式通关能力，而是把早期训练中的错误转化为可复盘、可惩罚、可学习的纠错样本。

## 核心立场

回溯是训练脚手架，不是模型能力。

正式能力必须满足：

```text
只能使用玩家当前可见信息；
不能读取隐藏抽牌顺序、隐藏随机数或敌人未来动作；
最终评测必须关闭回溯。
```

训练早期允许回溯，是为了降低探索成本，让模型更快看到“同一个局面下什么动作会失败、什么动作更合理”。但每次回溯都必须记录、扣分，并在训练后期逐步关闭。

## 为什么需要回溯训练

杀戮尖塔类游戏的学习难点不只是规则复杂，还包括：

- 一次错误可能几十分钟后才体现为死亡。
- 关键动作稀疏，例如药水时机、斩杀判断、低血量防守。
- 同一局面下可行动作很多，错误路径会浪费大量采集时间。
- LLM API 本身不会因为多玩几局自动永久变强，必须把纠错过程沉淀成数据。

回溯训练的价值在于：当模型在一个明确局面中犯错时，系统可以把这个错误变成结构化样本，而不是只得到一条“最后输了”的粗糙结果。

## 公平信息边界

所有进入模型输入和训练特征的数据必须分级。

| 分级 | 允许用途 | 例子 |
| --- | --- | --- |
| visible | 可进入 prompt、训练和正式评测 | 当前手牌、敌人当前意图、血量、能量、药水、遗物、弃牌堆、消耗区、抽牌堆公开信息 |
| derived_visible | 可进入 prompt、训练和正式评测 | 基于可见信息计算的最大伤害、最大格挡、生存差值、斩杀差值 |
| debug_only | 只允许调试，不进入训练和评测 | 内部对象状态、恢复校验细节、实验性存档指纹 |
| forbidden_for_training | 禁止进入训练和评测 | 真实下一张牌、隐藏 RNG、敌人未显示的未来动作、读档后才知道的结果 |

关键原则：

- 可以告诉模型“抽牌堆剩 10 张，其中公开可见防御牌约 3 张”。
- 不能告诉模型“下一张牌一定是 Defend”。
- 可以告诉模型“敌人当前意图是攻击 18”。
- 不能告诉模型“敌人下回合一定会 Buff”，除非游戏当前 UI 已经公开。

## 运行模式

建议项目明确区分四种模式。

| 模式 | 用途 | 回溯 | 是否计入正式成绩 |
| --- | --- | --- | --- |
| no_rewind | 最终演示和评测 | 禁用 | 是 |
| guarded_play | 日常演示 | 只允许执行前安全拒绝 | 可单独标记 |
| training_rewind | 训练早期纠错 | 允许回到检查点 | 否 |
| debug_oracle | 调试环境能力 | 可使用内部信息验证 | 否 |

所有 run 必须带模式标记：

```json
{
  "evaluation_mode": "training_rewind",
  "rewind_enabled": true,
  "rewind_count": 2,
  "hidden_info_used": false
}
```

只有 `evaluation_mode = no_rewind` 且 `rewind_count = 0` 的结果，才能作为真实能力结果。

## 回溯训练流程

推荐流程：

```text
1. 回合开始保存可见 checkpoint。
2. 模型在可见状态下选择动作。
3. 本地环境执行动作并观察结果。
4. 如果触发回溯条件，恢复到同一 checkpoint。
5. 系统只反馈基于可见信息的失败原因。
6. 模型重新选择动作。
7. 记录错误动作、失败原因、修正动作、惩罚和后续结果。
```

反馈可以包含：

```text
结束回合后会受到 18 点伤害，当前有效血量只有 12。
你没有使用可用的格挡药水。
当前敌人可见血量为 6，本回合存在可见斩杀动作。
```

反馈不能包含：

```text
下回合你会抽到防御牌。
敌人下回合会攻击 24。
刚才读档验证后发现另一条分支能赢。
```

## 回溯触发条件

第一版只使用确定性、可解释的触发条件。

| 条件 | 处理 |
| --- | --- |
| 执行 `end_turn` 会立即死亡 | 强制拒绝或回溯 |
| 明确可见斩杀被错过，且错误动作导致风险显著升高 | 可回溯 |
| 药水能避免死亡，但模型选择不用 | 可回溯 |
| 低血量精英/Boss 战出现明显资源误用 | 可回溯 |
| 普通战斗只是少打少挡几滴 | 不回溯 |
| 需要隐藏未来信息才能判断的错误 | 不回溯 |

触发原因使用标准标签：

```text
lethal_after_end_turn
missed_visible_lethal
ignored_survival_potion
avoidable_boss_death
excessive_visible_hp_loss
invalid_action_loop
```

## 惩罚和退火

回溯必须付费。否则模型会学会依赖试错。

### 建议惩罚

| 类型 | 基础惩罚 |
| --- | --- |
| 执行前虚拟拒绝 | -0.02 |
| 回到回合开始 | -0.10 |
| 回到战斗开始 | -0.50 |
| 同一战斗连续回溯 | 递增惩罚 |
| 回溯后仍死亡 | 额外惩罚 |

递增公式：

```text
penalty = base_penalty * (1 + rewind_count_in_battle)
```

### 阶段安排

| 阶段 | 目标 | 限制 |
| --- | --- | --- |
| early | 快速收集纠错样本 | 每场最多 3 次回溯 |
| middle | 减少依赖 | 每场最多 1 次回溯，惩罚加倍 |
| late | 接近真实环境 | 禁用恢复，只保留执行前安全拒绝 |
| final | 真实评测 | no_rewind，回溯和隐藏信息全部禁用 |

## 数据沉淀

LLM 的上下文不是长期记忆。不要把大量历史塞回 prompt。真正有价值的是把纠错过程写成短样本。

推荐记录格式：

```json
{
  "checkpoint_id": "act1_floor6_round2_turn_start",
  "evaluation_mode": "training_rewind",
  "visible_state_hash": "sha256:...",
  "bad_action": {
    "action": "end_turn"
  },
  "failure_reason": "lethal_after_end_turn",
  "feedback_to_model": "结束回合后会受到 18 点伤害，当前有效血量只有 12。",
  "corrected_action": {
    "action": "use_potion",
    "slot": 0
  },
  "penalty": -0.1,
  "rewind_count_in_battle": 1,
  "outcome_delta": {
    "hp_saved": 12,
    "enemy_killed": false
  },
  "hidden_info_used": false
}
```

这类样本可以用于：

- 更新规则 guardrail。
- 训练候选动作评分模型。
- 构造偏好数据。
- 复盘 LLM 错误类型。
- 后期做奖励模型或强化学习。

## 与 LLM 的关系

LLM 可以参与纠错，但不能控制回溯环境。

允许：

```json
{
  "meta_action": "request_replan",
  "reason": "当前计划会导致死亡，需要重新评估防御和药水"
}
```

不允许：

```text
LLM 自己关闭游戏、重启游戏、读取隐藏状态、选择读档点。
```

环境控制权必须留在本地 `RewindController`，由它决定是否允许回溯、如何恢复、如何验证、如何记录惩罚。

## 可能出现的问题

### 1. 模型依赖回溯

如果回溯成本太低，模型可能学会激进乱试。

控制方式：

- 每场限制次数。
- 递增惩罚。
- 训练后期关闭。
- 正式评测必须 `no_rewind`。

### 2. 信息泄漏

如果把真实抽牌顺序、隐藏随机数或未来敌人动作放进训练，模型会学到不公平策略。

控制方式：

- 建立 `visible / debug_only / forbidden_for_training` 数据分级。
- 训练前过滤字段。
- run 日志记录 `hidden_info_used`。

### 3. 回溯恢复不一致

如果外部读档没有回到同一局面，纠错样本会污染数据。

控制方式：

- checkpoint 保存 fingerprint。
- 恢复后必须校验手牌、血量、敌人、意图、牌堆数量。
- 校验失败则记录 `restore_failed`，不生成训练样本。

### 4. LLM 不会永久学习

API LLM 不会因为当前局多试几次就永久变强。

控制方式：

- prompt 只放当前可见状态和必要反馈。
- 完整历史只进本地日志。
- 纠错样本用于后续训练和规则改进。

### 5. 成绩不可比较

如果把回溯训练成绩和无回溯成绩混在一起，会误判模型能力。

控制方式：

- run 必须标记模式。
- 报告中区分 `training_rewind` 和 `no_rewind`。
- 只有 `no_rewind` 结果进入正式通过率。

## 对外表述建议

推荐表述：

```text
我们使用回溯训练作为早期纠错机制。它不向模型暴露隐藏信息，也不计入最终评测成绩。每次回溯都会记录原因和惩罚，用于生成更高质量的纠错样本。随着训练推进，回溯次数会逐步减少，最终在 no_rewind 模式下验证模型真实能力。
```

不推荐表述：

```text
让 AI 通过反复读档打赢。
让模型看后续抽牌再选择。
用回溯保证通关。
```

## 当前推荐执行

短期先做三件事：

1. 在 LLM 执行前加入虚拟拒绝：明显会死、明显错过可见斩杀时不执行。
2. 建立回溯样本日志：保存错误动作、失败原因、纠正动作、惩罚。
3. 做外部读档验证，但外部读档数据先标记为 `training_rewind`，不计入正式成绩。

长期目标：

```text
训练阶段允许模型低成本犯错；
数据阶段把错误变成样本；
评测阶段关闭回溯，验证模型真实能力。
```
