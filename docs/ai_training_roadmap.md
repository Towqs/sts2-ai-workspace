# STS2 AI 训练大纲、发展路线与当前状态

更新时间：2026-04-29

本文是项目记忆入口。目标是把高层 AI 训练设想落成可执行工程路线，避免后续忘记当前进度、技术取舍和下一步优先级。

## 一句话目标

基于 STS2MCP 的本地 API 和数据采集能力，先训练一个稳定完成第一章的半自动 AI，再逐步扩展到全流程自动、可评测、可迭代的《杀戮尖塔2》AI 玩家。

## 总体大纲

这个项目最终需要六个核心模块：

| 模块 | 含义 | 当前状态 |
| --- | --- | --- |
| 状态表示 | 把血量、能量、手牌、敌人、遗物、药水、地图、奖励等转换成模型输入 | 已有基础战斗/宏观编码，但维度和语义还偏粗 |
| 动作空间 | 把出牌、用药、结束回合、选路、选卡、事件、营火、商店变成可执行动作 | 已通过 STS2MCP API 打通，战斗动作建模仍需升级 |
| 环境接口 | 让外部程序读取状态并执行动作 | 已有 C# Mod + HTTP/MCP 接口 |
| 数据采集 | 记录人类/AI 的状态、动作、结果、run 质量 | 已有，但需要继续加固完整性和 schema |
| 行为克隆 | 用人类数据先训练基础策略 | 已有战斗 BC 和宏观 BC 的 MLP 基线 |
| 强化学习 | 在稳定环境中用 reward 继续优化策略 | 暂不具备条件，属于中长期目标 |

## 关键判断

### 1. 现阶段不应该优先做纯代码模拟器

纯代码模拟器是强化学习的理想终局，但现在成本过高。STS2 的卡牌、遗物、事件、怪物、费用、状态联动复杂，重写模拟器会产生大量偏差。模拟器一旦和真实游戏逻辑不一致，模型会学到错误策略。

当前更务实的路线是：

1. 继续使用 STS2MCP 对真实游戏做低速但准确的交互。
2. 稳定数据采集和自动评测。
3. 训练并验证第一章可用策略。
4. 后续再考虑局部模拟器或战斗重放器。

### 2. 现阶段不应该直接上 PPO

PPO/Actor-Critic 需要稳定环境、完整 reward、自动评测和大量 episode。当前主要短板是数据闭环与评测闭环，而不是算法名词。

上 PPO 前至少需要：

- 自动跑局不频繁卡死。
- 每局有完整 run 指标。
- reward 可以从日志稳定计算。
- 有可靠 baseline 做对比。
- 至少几千到几万条高质量动作样本。

### 3. 下一步核心是工程化，而不是扩大模型

当前模型是 MLP baseline，足够暴露数据问题。先把数据、动作、评测稳定下来，再换 Transformer 或 Actor-Critic 才有意义。

## 发展路线

### Phase 0：当前 baseline

已完成内容：

- STS2MCP Mod 接入。
- HTTP/MCP 状态读取和动作执行。
- 人类/AI 数据源拆分：`Human/Combat`、`Human/Macro`、`AI/Combat`、`AI/Macro`。
- 控制台支持采集开关、AI 开关、宏观开关、商店保护、训练、数据打包。
- 战斗 BC 和宏观 BC 训练脚本。
- LLM Advisor / Combat Auto 基础接入。

当前限制：

- `RL_DataCollector.cs` 仍然过大，职责混在一起。
- 战斗训练动作是 `play_card_卡牌ID`，不能充分表达目标选择。
- 宏观 BC 样本少，不能全权信任。
- 缺少系统化评测，不知道每次改动是否真的增强。
- `battle_start` / `turn_start` / `battle_end` 仍依赖 hook 稳定性。

### Phase 1：数据闭环加固

目标：每一局数据可复盘、可训练、可判断质量。

任务：

- 拆分 `RL_DataCollector.cs`：
  - `RLDataPaths`
  - `RLRunSession`
  - `RLStateBuilder`
  - `RLRecordWriter`
  - `RLCombatHooks`
  - `RLMacroHooks`
  - `RLShopSnapshot`
- 所有记录增加 `schema_version`。
- 明确 run 生命周期：`game_start`、`game_resume`、`battle_start`、`turn_start`、`action`、`turn_end`、`battle_end`、`run_end`。
- 控制台增加更明确的数据健康检查。
- 训练脚本只依赖统一 schema，减少临时兼容逻辑。

完成标准：

- 一局从开局到死亡/通关能形成完整 run 记录。
- 控制台能明确指出缺失的关键记录。
- 训练脚本对坏数据会跳过并计数，不静默吞掉。

### Phase 2：动作空间重构

目标：从“预测卡牌 ID”改成“对合法动作打分”。

当前战斗模型问题：

- `play_card_STRIKE` 无法表达打哪个敌人。
- 同一张牌不同目标价值不同。
- 动作字典和卡牌 ID 绑定，泛化弱。
- Action Masking 不够自然。

新设计：

```text
输入 = 当前状态 + 候选动作特征
输出 = 候选动作分数
```

候选动作：

- `play_card(hand_index, target_id)`
- `use_potion(slot, target_id)`
- `end_turn`

候选动作特征：

- 动作类型。
- 卡牌/药水 ID。
- 费用。
- 目标 HP、格挡、意图。
- 预估伤害、预估格挡。
- 是否斩杀。
- 是否防止致死。
- 是否保留费用。

执行策略：

1. 当前状态枚举所有合法动作。
2. 模型给每个候选动作打分。
3. 安全规则过滤明显错误动作。
4. 选最高分动作执行。
5. 执行后重新获取状态，避免手牌索引漂移。

完成标准：

- 模型能区分同一张攻击牌打不同敌人的价值。
- 非法动作不会进入候选集。
- `end_turn` 只有在合理时才被允许高分执行。

### Phase 3：评测系统

目标：每次改动后能知道 AI 是变强还是变弱。

每局记录指标：

| 指标 | 含义 |
| --- | --- |
| max_floor | 最高楼层 |
| max_act | 最高章节 |
| combats | 战斗数 |
| wins/losses | 战斗胜负 |
| elite_wins | 精英胜利数 |
| boss_wins | Boss 胜利数 |
| average_hp_loss | 平均掉血 |
| average_turns | 平均战斗回合 |
| invalid_actions | 非法动作次数 |
| stuck_count | 卡死/等待次数 |
| policy_name | 策略来源 |
| model_version | 模型版本 |

需要比较的 baseline：

- 规则引擎。
- 战斗 BC。
- 规则 + BC 混合。
- LLM Combat Auto。
- 后续 PPO 策略。

完成标准：

- 控制台能显示最近 N 局的策略表现。
- 每次模型更新能看到和上一版的差异。
- 失败原因能分类：战斗死亡、宏观误选、事件失误、卡死、非法动作。

### Phase 4：宏观策略混合化

目标：宏观不依赖单一 BC，而是规则、模型和 LLM 分层。

当前判断：

- 宏观样本 258 条，不足以单独训练可靠策略。
- 地图、事件、商店的风险比普通战斗更高。
- 商店购买应该继续有独立开关。

建议策略：

| 场景 | 主策略 | 辅助策略 |
| --- | --- | --- |
| 地图 | 规则评分 | BC 参考 |
| 选卡 | 角色卡牌评分 | BC 高置信覆盖 |
| 奖励 | 固定流程 | BC 不参与 |
| 事件 | 白名单/黑名单 | LLM Advisor |
| 营火 | HP 阈值规则 | 角色配置 |
| 商店 | 默认建议不购买 | 用户开关后执行保守购买 |

完成标准：

- 宏观 AI 不会因为少量样本做高风险动作。
- 商店和事件的执行都有保护。
- 控制台能显示宏观动作原因。

### Phase 5：LLM 作为裁判和解释层

目标：让 LLM 用在高价值决策点，而不是替代全部执行逻辑。

LLM 适用场景：

- Boss/精英复杂回合。
- 选卡解释。
- 事件风险判断。
- 商店购买建议。
- 失败复盘。

约束：

- LLM 只能选择候选动作 ID。
- 不允许自由生成 API 参数。
- 每次 LLM 决策必须保存：
  - compact state
  - candidate actions
  - raw response
  - parsed action
  - execution result

完成标准：

- LLM 决策可复盘。
- LLM 不能越过动作校验。
- LLM 成本和调用频率可控。

### Phase 6：强化学习准备

目标：在已有 baseline 和评测体系上，再引入 PPO/Actor-Critic。

前置条件：

- 真实环境自动跑局稳定。
- reward 从日志中可复现。
- 策略评测可自动对比。
- 战斗候选动作评分已完成。
- 至少具备第一章稳定 baseline。

初始 reward 设计：

| 事件 | 奖励 |
| --- | --- |
| 普通战斗胜利 | +10 |
| 精英胜利 | +30 到 +50 |
| Boss 胜利 | +100 到 +200 |
| 通关 | +1000 以上 |
| 死亡 | -100 |
| 掉血 | -1 / 点 |
| 非法动作 | -5 |
| 卡死 | -20 |
| 低血进入精英 | 负向 shaping |
| 斩杀敌人 | 小正奖励 |

注意：辅助奖励只能用于加速学习，不能压过通关目标，否则模型会刷局部收益。

## 当前不要做的事

- 不要马上重写完整纯代码模拟器。
- 不要在缺少评测时盲目换大模型结构。
- 不要让宏观 BC 全权控制事件和商店。
- 不要把 AI 自己失败样本默认混入训练。
- 不要把 API Key、模型权重、原始数据提交 Git。

## 工作原则

1. 先稳定采集，再扩大训练。
2. 先建立评测，再谈变强。
3. 先用规则保护，再让模型选择。
4. 先完成第一章 MVP，再扩展全角色全章节。
5. 每次策略更新必须能解释、能复盘、能回滚。
