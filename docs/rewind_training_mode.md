# 回溯训练模式设计稿

更新时间：2026-04-29

本文用于设计一个通用的战斗回溯能力。目标不是给某一个 LLM 写特殊外挂，而是把“失败前回到检查点、重新规划、扣分记录”做成训练环境能力，让 LLM、BC、RL 或后续自研模型都能使用。

## 一句话结论

回溯应该被设计成：

```text
环境工具，而不是模型能力。
```

模型可以请求回溯，也可以被系统强制回溯，但真正执行回溯、验证状态、记录惩罚，都必须由本地环境控制器负责。

## 用户观察

目前有一个重要经验：

```text
当一局快要输了时，直接关闭游戏，再重新打开，游戏有时会回到之前的存档点；
重新进入后，手牌和后续出牌随机性看起来仍然一致。
```

这个现象非常有价值，因为它说明 STS2 的存档和随机数可能具备以下特征：

- 游戏不会每个细小动作都立即覆盖存档。
- 继续游戏可能能恢复到战斗内某个较早状态。
- 同一个 run seed / RNG 状态下，抽牌、敌人行动、随机结果可能可复现。

但这还不能直接等同于“我们已经有稳定回溯接口”。工程上必须先验证：

- 关闭/重开后具体回到哪个点：战斗开始、回合开始、上一次 autosave，还是某个 UI 状态。
- 出牌后是否保存；结束回合后是否保存；用药后是否保存。
- 不同事件、奖励、Boss、精英、抽牌、生成牌场景是否稳定。
- 是否存在某些操作会立刻写盘，导致无法回退。

所以第一阶段要把它当作“可验证的外部回档能力”，不能假设百分百可靠。

## 回溯的三种层级

### 1. 虚拟回溯

不真的恢复游戏，只是在执行危险动作前拦截。

例子：

```text
LLM 想结束回合
本地评估发现结束后会死
系统拒绝执行 end_turn
重新要求 LLM 选择防御、药水或斩杀动作
```

优点：

- 现在就能做。
- 不需要改 Mod。
- 对 LLM 演示模式最有用。

缺点：

- 只能防止明显坏动作。
- 如果坏动作已经执行了，不能撤销。

### 2. 外部读档回溯

利用“关闭游戏再重新打开可以回到较早存档”的现象。

流程：

```text
记录回合开始 fingerprint
执行动作
发现必死或严重错误
关闭游戏进程
重新启动游戏
点击继续/等待自动恢复
读取当前状态
验证是否回到 checkpoint 附近
重新让模型决策
记录一次 rewind 惩罚
```

优点：

- 可能不需要立即改 C# Mod。
- 如果 autosave 行为稳定，可以很快形成训练工具。

缺点：

- 慢。
- 依赖游戏存档时机。
- 需要能自动启动游戏、等待 Mod API、进入继续游戏。
- 回到的位置可能不是我们想要的精确回合。

### 3. Mod 级精确快照

在 C# Mod 内实现真正的 `save_checkpoint / restore_checkpoint`。

理论上要保存：

- 玩家血量、格挡、能量。
- 手牌、抽牌堆、弃牌堆、消耗区。
- 敌人血量、格挡、意图、状态。
- 药水、遗物、能力、临时状态。
- 随机数状态。
- 战斗队列、回合数、触发器、遗物计数器。

优点：

- 最快。
- 最稳定。
- 最适合 RL 和分支搜索。

缺点：

- 难度最高。
- STS2 内部状态可能远比 API 暴露的 JSON 多。
- 漏一个计数器，就可能出现“看起来恢复了，实际规则错了”。

## 推荐路线

不要一开始就做 Mod 级完整快照。推荐顺序：

```text
P0：验证外部读档是否稳定
P1：先做虚拟回溯，防止 LLM 明显送死
P2：做外部读档回溯控制器
P3：把回溯记录接入训练数据和评分
P4：长期再做 Mod 级精确快照
```

这条路线可以最快让当前 LLM 受益，同时不把项目拖进完整游戏引擎重写。

## 通用工具抽象

回溯工具不应该绑定 LLM。建议抽象成：

```text
RewindController
```

核心接口：

```text
create_checkpoint(scope, state, metadata)
restore_checkpoint(checkpoint_id, method)
validate_restore(checkpoint_id, current_state)
record_rewind_event(reason, penalty, policy_name)
```

### checkpoint 类型

| 类型 | 说明 | 用途 |
| --- | --- | --- |
| battle_start | 战斗开始 | 分支搜索、Boss/精英训练 |
| turn_start | 玩家回合开始 | 最常用，适合纠正本回合错误 |
| action_before | 某个动作执行前 | 精确，但外部读档未必支持 |
| macro_node | 地图/奖励/事件选择前 | 宏观策略训练 |

### restore 方法

| 方法 | 含义 | 当前可行性 |
| --- | --- | --- |
| virtual_reject | 不恢复，只拒绝危险动作 | 立即可做 |
| reload_autosave | 关闭重启游戏，依赖游戏存档 | 需要验证 |
| mod_snapshot | C# Mod 内部恢复对象状态 | 长期目标 |

## 状态指纹

因为外部读档不一定精确回到同一点，所以必须做状态验证。每个 checkpoint 保存一个 fingerprint。

建议 fingerprint 包含：

```json
{
  "state_type": "monster",
  "act": 1,
  "floor": 6,
  "round": 2,
  "player": {
    "hp": 42,
    "block": 0,
    "energy": 3,
    "hand_ids": ["STRIKE", "BASH", "DEFEND"],
    "draw_pile_count": 8,
    "discard_pile_count": 5,
    "exhaust_pile_count": 0,
    "potions": ["FIRE_POTION"]
  },
  "enemies": [
    {
      "id": "JAW_WORM_0",
      "hp": 31,
      "block": 0,
      "intents": ["Attack 11"]
    }
  ]
}
```

恢复后检查：

- 是否同一战斗。
- 是否同一回合或可接受的较早回合。
- 手牌是否一致。
- 敌人血量和意图是否一致。
- 药水、遗物、牌堆数量是否一致。

如果验证失败，不能继续假装回溯成功。必须记录：

```text
restore_failed
```

然后停止自动训练或退回普通模式。

## 回溯触发条件

第一版建议使用确定性触发：

| 条件 | 是否回溯 |
| --- | --- |
| 执行 end_turn 后必死 | 必须回溯或拒绝 |
| 用错药水导致无法斩杀/无法保命 | 可回溯 |
| 错过明确斩杀，下一回合进入危险 | 可回溯 |
| Boss/精英战中低血量且有可用替代路线 | 可回溯 |
| 普通小怪损失过多血量 | 训练模式可回溯 |
| 只是少打几滴伤害 | 不回溯 |

触发原因建议标准化：

```text
lethal_after_end_turn
missed_lethal
avoidable_death
bad_potion_timing
excessive_hp_loss
invalid_action_loop
```

## 分数和惩罚

回溯不能免费，否则模型会学会乱试。建议把回溯作为“技能”，但使用会扣分。

### 基础惩罚

| 回溯类型 | 建议扣分 |
| --- | --- |
| virtual_reject | -0.02 |
| turn_start rewind | -0.10 |
| battle_start rewind | -0.50 |
| 多次连续回溯 | 每次递增 |
| 回溯后仍死亡 | 额外大扣分 |

### 递增惩罚

同一场战斗第 N 次回溯：

```text
penalty = base_penalty * (1 + rewind_count_in_battle)
```

例如：

```text
第 1 次 turn_start rewind：-0.10
第 2 次 turn_start rewind：-0.20
第 3 次 turn_start rewind：-0.30
```

训练目标不是“允许无限试错”，而是让模型知道：

```text
回溯能救命，但不是免费。
```

## 与 LLM 的关系

当前 LLM 可以立刻使用回溯思想，但不应该直接控制游戏进程。

### 当前可做

LLM 输出动作前，本地系统先检查：

```text
这个动作会不会导致必死？
是否错过明确斩杀？
是否有药水能救命但没有用？
```

如果发现明显错误：

```text
拒绝执行 -> 把原因反馈给 LLM -> 重新选择候选动作
```

这就是虚拟回溯。

### 外部读档可用后

LLM 可以多一个元动作：

```json
{
  "meta_action": "request_rewind",
  "reason": "本回合前两步导致无法防住敌人攻击，请回到回合开始重新规划"
}
```

但这个元动作不能直接进入游戏 API。执行权在 `RewindController`：

```text
LLM 请求 -> 本地判断是否允许 -> 执行恢复 -> 验证状态 -> 重新决策 -> 记录扣分
```

### 不建议的做法

不建议让 LLM 自己决定“杀游戏进程、重启、点继续”。这会把环境控制、UI 操作和策略混在一起，后续很难训练和复盘。

## 与自研模型/RL 的关系

回溯功能对后续训练价值更大：

- 可以让模型在同一个战斗状态下尝试不同分支。
- 可以把“错误动作 -> 回溯 -> 正确动作”变成对比数据。
- 可以降低采集成本，不必每次死亡都从头开一局。
- 可以做 curriculum learning：先允许回溯，后逐步提高回溯惩罚，最后禁用回溯。

训练数据可以记录：

```json
{
  "checkpoint_id": "turn_001_act1_floor6_round2",
  "bad_action": {"action": "end_turn"},
  "failure_reason": "lethal_after_end_turn",
  "rewind_method": "reload_autosave",
  "retry_action": {"action": "use_potion", "slot": 0},
  "penalty": -0.1,
  "outcome_delta": {
    "hp_saved": 12,
    "enemy_killed": false
  }
}
```

这类数据比普通日志更有训练价值，因为它直接告诉模型：同一状态下哪个选择更差，哪个选择更好。

## 外部读档验证方案

需要先做一个小测试，不动训练逻辑。

### 测试 1：回合开始稳定性

步骤：

1. 进入战斗玩家回合。
2. 读取状态，保存 fingerprint A。
3. 打一张牌。
4. 关闭游戏。
5. 重新打开并继续。
6. 读取状态，得到 fingerprint B。
7. 比较 A 和 B。

通过标准：

- B 回到 A，或回到同一战斗更早的安全状态。
- 手牌、敌人、血量、牌堆数量一致。

### 测试 2：结束回合后稳定性

步骤：

1. 玩家回合开始保存 A。
2. 点击结束回合。
3. 在敌人行动后或新回合前关闭游戏。
4. 重新打开。
5. 比较恢复点。

目的：

- 判断 end_turn 是否会触发保存。
- 判断能否回到上一个玩家回合。

### 测试 3：药水和抽牌场景

分别测试：

- 使用药水后关闭。
- 打抽牌牌后关闭。
- 打生成牌后关闭。
- 打消耗牌后关闭。

目的：

- 验证随机性是否完全复现。
- 验证临时生成牌、消耗区、药水槽是否可恢复。

### 测试 4：Boss/精英场景

至少测试：

- 普通小怪。
- 精英。
- Boss。

目的：

- 确认不同战斗类型存档行为一致。

## 实现拆分

### P0：文档和验证脚本

新增：

```text
AI_Training/rewind_probe.py
```

用途：

- 保存当前状态 fingerprint。
- 等待用户手动执行动作/关闭重开。
- 再读取状态并比较。
- 输出是否恢复到 checkpoint。

### P1：虚拟回溯接入 LLM

在 `llm_agent.py` 中：

- LLM 选择动作后，执行前计算风险。
- 如果动作导致明显死亡或错过斩杀，拒绝执行。
- 重新 prompt LLM，并加入拒绝原因。
- 记录 `virtual_reject`。

### P2：外部读档控制器

新增：

```text
AI_Training/rewind_controller.py
```

能力：

- 保存 checkpoint fingerprint。
- 停止 AI/LLM 自动动作。
- 关闭游戏进程。
- 重新启动游戏。
- 等待 Mod API 恢复。
- 验证状态。
- 恢复成功后重新交给策略。

### P3：训练日志接入

新增：

```text
RL_Datasets/Rewind/rewrite_events_YYYY-MM-DD.jsonl
```

每次回溯记录：

- checkpoint。
- 原动作。
- 失败原因。
- 恢复方法。
- 新动作。
- 惩罚。
- 最终结果。

### P4：前端控制

控制台增加：

- 回溯训练模式开关。
- 当前 checkpoint 状态。
- 本战斗回溯次数。
- 最近一次回溯原因。
- 是否允许外部重启游戏。
- 回溯惩罚配置。

### P5：Mod 级精确快照

长期研究：

- C# 内部对象快照。
- RNG 状态捕获。
- 战斗队列恢复。
- 遗物/能力计数器恢复。

只有外部读档不稳定，或训练速度成为瓶颈时，再推进这一阶段。

## 风险和边界

### 风险

- 外部读档可能不是每次都回到同一个点。
- 强杀游戏可能损坏当前 run 存档，需要先备份。
- 自动重启游戏会慢，不适合大量 RL。
- 如果验证不严格，模型会学到基于错误状态的策略。

### 边界

- 回溯训练模式是训练和调试工具，不应该混入“正常无回溯演示成绩”。
- 如果要展示无作弊能力，应关闭回溯，只保留虚拟拒绝和安全 guardrail。
- 回溯次数必须进入日志和评分，不能隐藏。

## 当前推荐判断

这个功能值得做，但优先级要拆开：

1. **立刻做虚拟回溯**：对当前 LLM 最有用，能阻止明显送死。
2. **马上验证外部读档**：因为用户已经观察到可能可行，验证成本低。
3. **验证通过后做外部读档控制器**：先服务演示和数据采集。
4. **最后再考虑 Mod 级快照**：只有当外部读档不够稳定或太慢时才值得投入。

最终目标是：

```text
模型可以犯错，但每次犯错都要付出可记录、可训练、可逐步减少的代价。
```
