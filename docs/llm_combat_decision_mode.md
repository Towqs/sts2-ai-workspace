# LLM 战斗决策模式设计稿

更新时间：2026-04-29

本文用于讨论下一版 LLM 战斗 AI。核心目标不是让大模型裸算游戏规则，而是让本地代码负责可计算内容，让 LLM 负责高层判断、取舍和解释。

## 一句话结论

合理模式是：

```text
真实游戏状态 -> 本地战斗计算器 -> 候选动作/局势摘要 -> LLM 判断 -> 本地校验/兜底 -> 执行 -> 记录复盘
```

LLM 是“大脑”，但手牌伤害、最大格挡、斩杀差值、药水收益、死亡风险这些能被代码算出来的内容，都应该先由代码算好再喂给 LLM。

## 为什么不能只靠默认脚本

杀戮尖塔的随机性很强：

- 每回合手牌不同。
- 卡牌奖励不同。
- 药水、遗物、事件、敌人组合不同。
- 同一张牌在不同敌人意图下价值不同。

所以默认策略不能写成固定出牌顺序。默认策略应该写成“判断原则”，例如：

- 能斩杀优先斩杀。
- 会死时优先生存。
- Boss/精英战不要过度省药。
- 敌人不攻击时优先输出或成长。
- 当前回合有高价值 0 费牌时不要提前结束。

这些原则由 LLM 理解，但原则需要建立在本地计算出的局势指标上。

## 分工边界

### 代码负责计算

代码适合做确定性、可验证、可复盘的计算：

| 计算项 | 含义 |
| --- | --- |
| incoming_damage | 敌人本回合预期总攻击 |
| current_block | 当前已有格挡 |
| survival_gap | 本回合距离死亡还差多少防御 |
| hand_max_damage | 当前手牌在能量约束下的最大即时伤害估计 |
| hand_max_block | 当前手牌在能量约束下的最大即时格挡估计 |
| lethal_gap | 敌人剩余有效血量与当前最大伤害之间的差距 |
| potion_lethal_bridge | 哪些药水能补足斩杀差值 |
| potion_survival_bridge | 哪些药水能避免死亡或显著减伤 |
| scaling_value | 力量、敏捷、能力牌、持续伤害类收益 |
| legal_candidates | 当前真实可执行动作列表 |

### 战斗计算指标定义

第一版建议把指标拆成四类，避免混在一起：

| 指标 | 作用 | 备注 |
| --- | --- | --- |
| turn_max_damage | 当前回合最大即时伤害 | 只看这一回合能实际打出的攻击/伤害牌 |
| turn_max_block | 当前回合最大即时格挡 | 只看这一回合能实际获得的格挡 |
| lethal_bridge | 距离斩杀还差多少 | 用来判断伤害药水、能量药水、易伤是否能补齐 |
| survival_bridge | 距离活下来还差多少 | 用来判断格挡药水、虚弱、减力量、技能生成是否必要 |
| scaling_damage_value | 持续输出收益 | 力量、易伤、持续伤害、能力牌等 |
| scaling_defense_value | 持续防御收益 | 敏捷、金属化、持续格挡、减伤等 |
| draw_or_energy_value | 抽牌/能量收益 | 判断能量药水、抽牌牌、生成牌药水是否能打开回合 |
| risk_level | 当前危险等级 | safe / pressured / lethal_risk |

即时伤害和持续收益要分开。比如力量药水可能这一手没有直接造成 20 点伤害，但在 Boss 战前期价值很高；火焰药水则更适合补斩杀差值。

### LLM 负责判断

LLM 适合做上下文判断：

- 当前回合应该斩杀、保命、启动成长，还是省资源。
- Boss/精英战是否该提前使用力量、敏捷、能力类药水。
- 多个合法动作都可用时，哪个更符合当前战斗节奏。
- 卡牌奖励、路线、营火、事件这类高层取舍。
- 解释“为什么这么做”，方便前端展示和复盘。

### 本地兜底负责安全

执行前必须校验：

- LLM 选择的动作是否来自合法候选。
- card_index、target、potion_slot 是否真实存在。
- LLM 是否提前结束了还有明显收益的回合。
- LLM 是否在濒死时忽略了可用防御/药水。
- LLM 是否浪费了关键药水。

校验失败时，本地代码可以自动改成更安全的候选动作，并记录 `guardrail` 原因。

## 决策流程

### 1. 读取状态

从 STS2MCP/API 读取当前状态：

- 玩家血量、格挡、能量。
- 手牌、费用、是否可打、卡牌描述。
- 敌人血量、格挡、意图、状态。
- 药水、遗物、抽牌堆、弃牌堆。
- 当前楼层、Act、战斗类型、Boss/精英标记。

### 2. 枚举合法动作

继续沿用当前候选动作结构：

```text
play_card(hand_index, target_id)
use_potion(slot, target_id?)
end_turn
proceed / reward / event / campfire 等宏观动作
```

LLM 不应该构造游戏不存在的动作。即使使用“动作参数模式”，最后也要映射回合法候选。

### 3. 本地生成战斗评估

新增一个战斗评估层，输出类似：

```json
{
  "priority": "lethal_or_survive",
  "incoming_damage": 18,
  "current_block": 0,
  "hp_after_no_action": 7,
  "hand_max_damage": 24,
  "hand_max_block": 15,
  "lowest_enemy_effective_hp": 22,
  "lethal_gap": -2,
  "survival_gap": 3,
  "potion_opportunities": [
    {
      "slot": 1,
      "kind": "damage",
      "reason": "补足斩杀",
      "estimated_value": 20,
      "trigger": "lethal_bridge"
    }
  ]
}
```

这份摘要是喂给 LLM 的重点。LLM 不需要自己从一堆卡牌文本里硬算，而是基于摘要做判断。

### 4. LLM 选择动作

建议保留两种模式：

| 模式 | 用途 | 约束 |
| --- | --- | --- |
| candidate_id | 演示和稳定通关 | 只能选候选动作 ID |
| catalog_args | 更像自然决策 | 可以输出动作参数，但必须映射回候选动作 |

演示模式默认使用 `candidate_id`，因为稳定性最重要。

### 5. 本地校验和执行

执行前走 guardrail：

- 如果代码已经确认能斩杀，LLM 不能无理由结束回合。
- 如果结束回合会死，LLM 必须使用可用防御、药水或攻击斩杀。
- 如果药水能补足 Boss/精英斩杀差值，应允许积极使用。
- 如果普通战斗安全且药水收益低，应倾向保留。
- 如果 LLM 参数错误，自动映射到最接近的合法候选。

### 6. 记录复盘

每次 LLM 决策都记录：

- 原始状态摘要。
- 本地计算指标。
- 候选动作列表。
- LLM 原始回复。
- 最终执行动作。
- 是否触发 guardrail。
- 执行结果。

这样后面才能知道是 LLM 判断错、代码计算错、状态读取错，还是执行映射错。

## 牌区摘要

LLM 输入不应该保留上一回合完整流水账，但不能丢失牌区信息。杀戮尖塔的战斗不是只看当前手牌，还要看抽牌堆、弃牌堆和消耗区。

### 需要保留的区域

| 区域 | 输入方式 | 用途 |
| --- | --- | --- |
| hand | 保留当前手牌明细 | 当前回合决策核心 |
| draw_pile | 保留数量和分类摘要，必要时保留关键牌 | 判断下回合质量、抽牌收益、是否即将洗牌 |
| discard_pile | 保留数量和分类摘要 | 判断洗牌后卡组质量 |
| exhaust_pile | 保留数量和关键牌 | 判断核心牌是否已经离场 |
| deck | 战斗外保留完整构筑摘要 | 用于奖励、商店、删牌、升级、路线 |

### 推荐摘要格式

```json
{
  "pile_summary": {
    "draw_count": 4,
    "discard_count": 8,
    "exhaust_count": 1,
    "draw_tags": {
      "attack": 2,
      "block": 1,
      "draw": 1,
      "power": 0,
      "status_or_curse": 0
    },
    "discard_tags": {
      "attack": 4,
      "block": 2,
      "status_or_curse": 2
    },
    "key_cards_in_draw": ["Shrug It Off"],
    "key_cards_exhausted": ["Offering"],
    "reshuffle_soon": true,
    "next_turn_quality": "mixed"
  }
}
```

### 输入裁剪原则

- 当前手牌必须保留卡牌效果标签和简短效果描述。
- 抽牌堆和弃牌堆默认只给摘要，不给完整列表。
- 如果抽牌堆很少，或里面有关键牌，可以给关键牌名。
- 消耗区只保留关键牌，不展开所有普通牌。
- 上一回合完整出牌顺序不进入 LLM prompt，只留在本地日志。

## 卡牌语义压缩

如果只给最大伤害和最大格挡，LLM 会看不到抽牌、弃牌、消耗、易伤、虚弱、力量、敏捷、能力牌等效果。因此当前手牌需要做结构化语义压缩。

建议每张当前手牌输入：

```json
{
  "index": 0,
  "name": "Bash",
  "cost": 2,
  "type": "attack",
  "can_play": true,
  "tags": ["damage", "vulnerable"],
  "damage_est": 8,
  "block_est": 0,
  "effect_note": "造成伤害并施加易伤",
  "value_note": "适合斩杀前铺易伤，或打 Boss/精英前置增伤"
}
```

第一版标签足够覆盖：

- `damage`
- `block`
- `draw`
- `discard`
- `exhaust`
- `vulnerable`
- `weak`
- `strength`
- `dexterity`
- `energy`
- `power`
- `retain`
- `generate_card`
- `status_or_curse`

这能让 LLM 理解“这张牌为什么重要”，但不会把历史和长文本塞进 prompt。

## 药水作为一次性技能

药水不应该只是普通动作。它应该被当成“消耗型技能”，显示在前端技能槽，也参与本地战斗评估。

### 药水分类

| 类型 | 例子 | 主要用途 |
| --- | --- | --- |
| direct_damage | 火焰药水等 | 补斩杀、清危险敌人 |
| block | 格挡药水等 | 防死、降低本回合损失 |
| strength | 力量药水等 | Boss/精英战提前启动输出 |
| dexterity | 敏捷药水等 | 长战斗提升防御质量 |
| energy | 能量药水等 | 打出关键牌组合 |
| card_generation | 生成攻击/技能/能力牌 | 找斩杀、找防御、找启动 |
| debuff | 虚弱、易伤、减力量 | 降低伤害或提高斩杀线 |
| utility | 其他特殊效果 | 按白名单策略处理 |

### 药水触发原则

优先用药的情况：

- 不用药水杀不死，用药水能杀死。
- 不用药水会死，或掉到极危险血线。
- Boss/精英战前几回合，成长类药水能明显提高胜率。
- 药水槽已满，当前药水价值一般，且后续可能捡新药。

倾向保留的情况：

- 普通小怪战斗已经安全。
- 药水不能改变本回合结果。
- 用药后仍不能斩杀、不能保命，也没有长期收益。

## 动态调用和强制改打法机制

不建议每个动作都无脑调用 LLM。调用频率应该由风险等级决定：

| 场景 | 调用策略 |
| --- | --- |
| 普通小怪，血量安全，局势简单 | 每回合开始调用一次，后续由本地策略执行 |
| 普通小怪，但出现抽牌、生成牌、药水、敌人变化 | 状态明显变化后重新调用 |
| 精英/Boss | 默认每个关键动作后重新评估，必要时重新调用 |
| 血量危险 | 强制重新调用，并切换为保命策略 |
| 发现斩杀机会 | 强制重新评估，优先斩杀 |
| 结束回合前 | 必须本地检查，不一定调用 LLM |

### 风险等级

建议第一版定义四级：

| 等级 | 条件 | 行为 |
| --- | --- | --- |
| safe | 预计掉血后仍高于 50% 最大血量，且不是精英/Boss | 少调用，省 token |
| pressured | 预计掉血后低于 50% 最大血量，或敌人攻击较高 | 每回合调用，必要时动作后调用 |
| danger | 预计掉血后低于 30% 最大血量，或本回合损失超过 25% 最大血量 | 强制保命倾向，允许用药 |
| lethal_risk | 结束回合会死，或下回合极可能死 | guardrail 可覆盖 LLM |

### 强制改打法触发条件

以下条件触发时，本地系统不应继续沿用普通打法：

- `hp_after_incoming <= 0`：结束回合会死。
- `hp_after_incoming <= max_hp * 0.30`：进入危险血线。
- `incoming_damage >= max_hp * 0.25`：本回合敌人爆发较高。
- `state_type in elite/boss`：精英或 Boss 战。
- `lethal_gap <= 0`：当前有明确斩杀。
- `potion_lethal_bridge == true`：药水能补足斩杀。
- `potion_survival_bridge == true`：药水能避免死亡或大幅减伤。
- `draw_or_generate_changed_hand == true`：抽牌/生成牌后手牌发生显著变化。
- `llm_validation_failed == true`：LLM 动作参数错误或候选不存在。

触发后策略切换：

```text
普通打法 -> 保命 / 斩杀 / Boss精英积极资源 / 重新评估
```

### Agent 循环模式

推荐使用受约束 Agent，而不是自由 Agent：

```text
Observe 当前状态
Evaluate 本地计算战斗、药水、牌区和风险
Plan LLM 判断当前策略意图
Choose 只能选择合法候选动作
Execute 本地执行
Check 本地校验是否需要覆盖、重试或结束回合
```

Agent 可以循环，但每一步都要经过候选动作和本地校验。

回溯训练模式是这个 guardrail 的进一步扩展：当动作已经造成明显坏结果时，可以通过虚拟拒绝、外部读档或未来 Mod 快照回到检查点，并把这次回溯作为训练惩罚记录。详细设计见 [`rewind_training_mode.md`](rewind_training_mode.md)。

## 本地计算器的第一版范围

第一版不需要做完整游戏模拟器。先做“单回合保守评估”：

### 必做

- 统计敌人本回合总攻击。
- 统计当前手牌可打攻击牌的最大即时伤害估计。
- 统计当前手牌可打防御牌的最大格挡估计。
- 计算最低敌人有效血量。
- 计算是否存在明显斩杀机会。
- 计算结束回合是否会死亡或进入危险血线。
- 给每个药水打标签：伤害、防御、成长、能量、生成牌、控制、未知。
- 生成当前手牌卡牌语义标签。
- 生成抽牌堆、弃牌堆、消耗区摘要。
- 根据血量、敌人攻击、Boss/精英、斩杀机会计算风险等级。

### 第二版再做

- 能量约束下的最佳出牌组合搜索。
- 力量/敏捷/易伤/虚弱对伤害和防御的修正。
- 能量药水带来的额外组合收益。
- 生成牌药水的期望收益。
- 多敌人优先级：召唤物、低血危险敌人、Boss 本体。
- 持续伤害、能力牌、成长牌的长期价值评分。
- 更精确的下回合手牌质量预估。

### 暂时不做

- 不重写完整 STS2 战斗模拟器。
- 不预测未来抽牌的精确分布。
- 不让 LLM 直接绕过候选动作执行。

## 推荐演示模式

为了能给别人演示，建议新增一个明确的配置：

```text
LLM 演示模式
```

默认行为：

- 动作选择使用 `candidate_id`。
- 温度设为 0 或 0.1。
- 启用战斗评估摘要。
- 启用药水技能评估。
- 启用危险回合兜底。
- 启用提前结束回合兜底。
- Boss/精英战药水使用更积极。
- 普通小怪安全状态减少 LLM 调用，降低 token 消耗。
- 血量进入危险线、出现斩杀或药水关键机会时强制重新评估。
- 每局保存 LLM 决策复盘。

演示目标：

```text
铁甲 / A0 / 连续 5 局中至少 4 局通过 Act1 Boss
```

这个目标比“某一局过了”更有意义，因为它能排除运气因素。

## 前端展示建议

前端应该把 LLM 的“正在理解游戏”展示出来，而不是只显示一堆日志：

- 顶部显示当前战斗优先级：斩杀 / 保命 / 启动 / 省资源。
- 药水显示成一次性技能槽，用掉后消失或变灰。
- 最近动作流显示一句人能看懂的话：
  - `使用火焰药水：补足 12 点斩杀差值`
  - `打出 Bash：先挂易伤，为本回合斩杀做准备`
  - `改写 LLM 决策：结束回合会死亡，改为使用格挡药水`
- 详细数据折叠，不要默认铺满页面。

## 实现拆分

### P1：战斗评估器

新增或扩展模块：

```text
AI_Training/combat_evaluator.py
```

输出 `combat_eval`：

- 敌人攻击。
- 最大伤害估计。
- 最大格挡估计。
- 斩杀差值。
- 生存差值。
- 危险等级。

### P2：牌区和卡牌语义评估

新增：

- 当前手牌语义标签。
- 抽牌堆、弃牌堆、消耗区摘要。
- 关键牌是否在抽牌堆或已消耗。
- 是否即将洗牌。

### P3：药水技能化

新增药水分类和收益评估：

- `classify_potion(potion)`
- `evaluate_potion_opportunity(state, potion, combat_eval)`
- 结果写入 LLM prompt 和前端动态。

### P4：LLM prompt、动态调用和 guardrail

在 `AI_Training/llm_agent.py` 中：

- 把 `combat_eval` 放进 LLM 输入。
- 把 `pile_summary` 和当前手牌语义放进 LLM 输入。
- 强化候选动作模式 prompt。
- 增加斩杀兜底、生存兜底、药水兜底。
- 按风险等级决定调用频率。

### P5：自动评测

## 需要继续讨论的问题

1. 演示模式里，guardrail 是否允许直接覆盖 LLM？建议允许，否则稳定性不够。
2. 药水使用要多激进？建议 Boss/精英积极，普通小怪保守。
3. 第一版最大伤害/最大防御是只做启发式，还是直接做能量约束组合搜索？建议先启发式，第二版再组合搜索。
4. LLM 每个动作都调用，还是只在关键节点调用？建议按风险等级动态调用，普通小怪少调用，危险/Boss/精英/状态变化后多调用。
5. 宏观决策是否也进入演示模式？建议进入，但先用保守规则兜底。

## 当前推荐路线

先实现 P1 + P2 + P3 + P4 的第一版，不急着做完整模拟器。

也就是：

```text
代码先算清楚当前回合能不能杀、能不能活、药水有没有关键价值；
再把当前手牌语义、抽牌堆/弃牌堆/消耗区摘要、风险等级喂给 LLM；
LLM 根据这些摘要选策略和动作；
本地校验保证它不会乱按。
```

这条路线最适合现在的项目状态：实现成本可控，能明显增强演示效果，也能保留后续升级空间。
