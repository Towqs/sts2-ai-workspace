# LLM 战斗决策模式设计稿

更新时间：2026-04-29

本文用于讨论下一版 LLM 战斗 AI。核心目标不是让大模型裸算游戏规则，而是让本地代码负责可计算内容，让 LLM 负责高层判断、取舍和解释。

## 一句话结论

合理模式是：

```text
真实游戏状态 -> 本地战斗计算器 -> 候选动作/局势摘要 -> LLM 判断 -> 本地校验/兜底 -> 执行 -> 记录复盘
```

LLM 是“大脑”，但手牌伤害、最大格挡、斩杀差值、药水收益、死亡风险这些能被代码算出来的内容，都应该先由代码算好再喂给 LLM。

## 为什么不能只靠默认脚本

杀戮尖塔的随机性很强：

- 每回合手牌不同。
- 卡牌奖励不同。
- 药水、遗物、事件、敌人组合不同。
- 同一张牌在不同敌人意图下价值不同。

所以默认策略不能写成固定出牌顺序。默认策略应该写成“判断原则”，例如：

- 能斩杀优先斩杀。
- 会死时优先生存。
- Boss/精英战不要过度省药。
- 敌人不攻击时优先输出或成长。
- 当前回合有高价值 0 费牌时不要提前结束。

这些原则由 LLM 理解，但原则需要建立在本地计算出的局势指标上。

## 分工边界

### 代码负责计算

代码适合做确定性、可验证、可复盘的计算：

| 计算项 | 含义 |
| --- | --- |
| incoming_damage | 敌人本回合预期总攻击 |
| current_block | 当前已有格挡 |
| survival_gap | 本回合距离死亡还差多少防御 |
| hand_max_damage | 当前手牌在能量约束下的最大即时伤害估计 |
| hand_max_block | 当前手牌在能量约束下的最大即时格挡估计 |
| lethal_gap | 敌人剩余有效血量与当前最大伤害之间的差距 |
| potion_lethal_bridge | 哪些药水能补足斩杀差值 |
| potion_survival_bridge | 哪些药水能避免死亡或显著减伤 |
| scaling_value | 力量、敏捷、能力牌、持续伤害类收益 |
| legal_candidates | 当前真实可执行动作列表 |

### 战斗计算指标定义

第一版建议把指标拆成四类，避免混在一起：

| 指标 | 作用 | 备注 |
| --- | --- | --- |
| turn_max_damage | 当前回合最大即时伤害 | 只看这一回合能实际打出的攻击/伤害牌 |
| turn_max_block | 当前回合最大即时格挡 | 只看这一回合能实际获得的格挡 |
| lethal_bridge | 距离斩杀还差多少 | 用来判断伤害药水、能量药水、易伤是否能补齐 |
| survival_bridge | 距离活下来还差多少 | 用来判断格挡药水、虚弱、减力量、技能生成是否必要 |
| scaling_damage_value | 持续输出收益 | 力量、易伤、持续伤害、能力牌等 |
| scaling_defense_value | 持续防御收益 | 敏捷、金属化、持续格挡、减伤等 |
| draw_or_energy_value | 抽牌/能量收益 | 判断能量药水、抽牌牌、生成牌药水是否能打开回合 |
| risk_level | 当前危险等级 | safe / pressured / lethal_risk |

即时伤害和持续收益要分开。比如力量药水可能这一手没有直接造成 20 点伤害，但在 Boss 战前期价值很高；火焰药水则更适合补斩杀差值。

### LLM 负责判断

LLM 适合做上下文判断：

- 当前回合应该斩杀、保命、启动成长，还是省资源。
- Boss/精英战是否该提前使用力量、敏捷、能力类药水。
- 多个合法动作都可用时，哪个更符合当前战斗节奏。
- 卡牌奖励、路线、营火、事件这类高层取舍。
- 解释“为什么这么做”，方便前端展示和复盘。

### 本地兜底负责安全

执行前必须校验：

- LLM 选择的动作是否来自合法候选。
- card_index、target、potion_slot 是否真实存在。
- LLM 是否提前结束了还有明显收益的回合。
- LLM 是否在濒死时忽略了可用防御/药水。
- LLM 是否浪费了关键药水。

校验失败时，本地代码可以自动改成更安全的候选动作，并记录 `guardrail` 原因。

## 决策流程

### 1. 读取状态

从 STS2MCP/API 读取当前状态：

- 玩家血量、格挡、能量。
- 手牌、费用、是否可打、卡牌描述。
- 敌人血量、格挡、意图、状态。
- 药水、遗物、抽牌堆、弃牌堆。
- 当前楼层、Act、战斗类型、Boss/精英标记。

### 2. 枚举合法动作

继续沿用当前候选动作结构：

```text
play_card(hand_index, target_id)
use_potion(slot, target_id?)
end_turn
proceed / reward / event / campfire 等宏观动作
```

LLM 不应该构造游戏不存在的动作。即使使用“动作参数模式”，最后也要映射回合法候选。

### 3. 本地生成战斗评估

新增一个战斗评估层，输出类似：

```json
{
  "priority": "lethal_or_survive",
  "incoming_damage": 18,
  "current_block": 0,
  "hp_after_no_action": 7,
  "hand_max_damage": 24,
  "hand_max_block": 15,
  "lowest_enemy_effective_hp": 22,
  "lethal_gap": -2,
  "survival_gap": 3,
  "potion_opportunities": [
    {
      "slot": 1,
      "kind": "damage",
      "reason": "补足斩杀",
      "estimated_value": 20,
      "trigger": "lethal_bridge"
    }
  ]
}
```

这份摘要是喂给 LLM 的重点。LLM 不需要自己从一堆卡牌文本里硬算，而是基于摘要做判断。

### 4. LLM 选择动作

建议保留两种模式：

| 模式 | 用途 | 约束 |
| --- | --- | --- |
| candidate_id | 演示和稳定通关 | 只能选候选动作 ID |
| catalog_args | 更像自然决策 | 可以输出动作参数，但必须映射回候选动作 |

演示模式默认使用 `candidate_id`，因为稳定性最重要。

### 5. 本地校验和执行

执行前走 guardrail：

- 如果代码已经确认能斩杀，LLM 不能无理由结束回合。
- 如果结束回合会死，LLM 必须使用可用防御、药水或攻击斩杀。
- 如果药水能补足 Boss/精英斩杀差值，应允许积极使用。
- 如果普通战斗安全且药水收益低，应倾向保留。
- 如果 LLM 参数错误，自动映射到最接近的合法候选。

### 6. 记录复盘

每次 LLM 决策都记录：

- 原始状态摘要。
- 本地计算指标。
- 候选动作列表。
- LLM 原始回复。
- 最终执行动作。
- 是否触发 guardrail。
- 执行结果。

这样后面才能知道是 LLM 判断错、代码计算错、状态读取错，还是执行映射错。

## 药水作为一次性技能

药水不应该只是普通动作。它应该被当成“消耗型技能”，显示在前端技能槽，也参与本地战斗评估。

### 药水分类

| 类型 | 例子 | 主要用途 |
| --- | --- | --- |
| direct_damage | 火焰药水等 | 补斩杀、清危险敌人 |
| block | 格挡药水等 | 防死、降低本回合损失 |
| strength | 力量药水等 | Boss/精英战提前启动输出 |
| dexterity | 敏捷药水等 | 长战斗提升防御质量 |
| energy | 能量药水等 | 打出关键牌组合 |
| card_generation | 生成攻击/技能/能力牌 | 找斩杀、找防御、找启动 |
| debuff | 虚弱、易伤、减力量 | 降低伤害或提高斩杀线 |
| utility | 其他特殊效果 | 按白名单策略处理 |

### 药水触发原则

优先用药的情况：

- 不用药水杀不死，用药水能杀死。
- 不用药水会死，或掉到极危险血线。
- Boss/精英战前几回合，成长类药水能明显提高胜率。
- 药水槽已满，当前药水价值一般，且后续可能捡新药。

倾向保留的情况：

- 普通小怪战斗已经安全。
- 药水不能改变本回合结果。
- 用药后仍不能斩杀、不能保命，也没有长期收益。

## 本地计算器的第一版范围

第一版不需要做完整游戏模拟器。先做“单回合保守评估”：

### 必做

- 统计敌人本回合总攻击。
- 统计当前手牌可打攻击牌的最大即时伤害估计。
- 统计当前手牌可打防御牌的最大格挡估计。
- 计算最低敌人有效血量。
- 计算是否存在明显斩杀机会。
- 计算结束回合是否会死亡或进入危险血线。
- 给每个药水打标签：伤害、防御、成长、能量、生成牌、控制、未知。

### 第二版再做

- 能量约束下的最佳出牌组合搜索。
- 力量/敏捷/易伤/虚弱对伤害和防御的修正。
- 能量药水带来的额外组合收益。
- 生成牌药水的期望收益。
- 多敌人优先级：召唤物、低血危险敌人、Boss 本体。
- 持续伤害、能力牌、成长牌的长期价值评分。

### 暂时不做

- 不重写完整 STS2 战斗模拟器。
- 不预测未来抽牌的精确分布。
- 不让 LLM 直接绕过候选动作执行。

## 推荐演示模式

为了能给别人演示，建议新增一个明确的配置：

```text
LLM 演示模式
```

默认行为：

- 动作选择使用 `candidate_id`。
- 温度设为 0 或 0.1。
- 启用战斗评估摘要。
- 启用药水技能评估。
- 启用危险回合兜底。
- 启用提前结束回合兜底。
- Boss/精英战药水使用更积极。
- 每局保存 LLM 决策复盘。

演示目标：

```text
铁甲 / A0 / 连续 5 局中至少 4 局通过 Act1 Boss
```

这个目标比“某一局过了”更有意义，因为它能排除运气因素。

## 前端展示建议

前端应该把 LLM 的“正在理解游戏”展示出来，而不是只显示一堆日志：

- 顶部显示当前战斗优先级：斩杀 / 保命 / 启动 / 省资源。
- 药水显示成一次性技能槽，用掉后消失或变灰。
- 最近动作流显示一句人能看懂的话：
  - `使用火焰药水：补足 12 点斩杀差值`
  - `打出 Bash：先挂易伤，为本回合斩杀做准备`
  - `改写 LLM 决策：结束回合会死亡，改为使用格挡药水`
- 详细数据折叠，不要默认铺满页面。

## 实现拆分

### P1：战斗评估器

新增或扩展模块：

```text
AI_Training/combat_evaluator.py
```

输出 `combat_eval`：

- 敌人攻击。
- 最大伤害估计。
- 最大格挡估计。
- 斩杀差值。
- 生存差值。
- 危险等级。

### P2：药水技能化

新增药水分类和收益评估：

- `classify_potion(potion)`
- `evaluate_potion_opportunity(state, potion, combat_eval)`
- 结果写入 LLM prompt 和前端动态。

### P3：LLM prompt 和 guardrail

在 `AI_Training/llm_agent.py` 中：

- 把 `combat_eval` 放进 LLM 输入。
- 强化候选动作模式 prompt。
- 增加斩杀兜底、生存兜底、药水兜底。

### P4：自动评测

增加 Act1 演示评测：

- 连续 N 局。
- 记录 Act1 Boss 通过率。
- 记录死亡楼层。
- 记录药水使用次数。
- 记录 guardrail 修正次数。
- 记录非法动作次数。

### P5：前端复盘展示

控制台显示：

- 当前优先级。
- 当前危险等级。
- 药水技能槽。
- 最近一条动作解释。
- 可展开的 LLM 详细决策。

## 需要继续讨论的问题

1. 演示模式里，guardrail 是否允许直接覆盖 LLM？建议允许，否则稳定性不够。
2. 药水使用要多激进？建议 Boss/精英积极，普通小怪保守。
3. 第一版最大伤害/最大防御是只做启发式，还是直接做能量约束组合搜索？建议先启发式，第二版再组合搜索。
4. LLM 每个动作都调用，还是只在关键节点调用？演示模式可以每步调用；长期版本应减少调用频率。
5. 宏观决策是否也进入演示模式？建议进入，但先用保守规则兜底。

## 当前推荐路线

先实现 P1 + P2 + P3，不急着做完整模拟器。

也就是：

```text
代码先算清楚当前回合能不能杀、能不能活、药水有没有关键价值；
LLM 根据这些摘要选动作；
本地校验保证它不会乱按。
```

这条路线最适合现在的项目状态：实现成本可控，能明显增强演示效果，也能保留后续升级空间。
