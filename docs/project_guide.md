# STS2 AI 工作区说明

STS2 AI Workspace 是一个面向 Slay the Spire 2 的本地 AI 工作区。它不是单个自动出牌脚本，而是一套“读游戏状态、采集数据、训练模型、让 AI 决策、再回收数据”的实验环境。

当前项目仍是半成品：控制台、数据采集、基础 BC 训练、候选动作评分、自训练和 LLM 接入已经跑通，但缺少大量高质量实战数据和高价值 self-play run，AI 的稳定性还需要继续验证。

仓库现在包含一个演示模型包 `AI_Training/ModelZoo/demo_local_20260430/`。它能让别人拉取仓库后直接试用基础战斗 AI、候选动作评分和宏观模型，但它仍然是早期模型，不代表稳定通关能力。

## 项目现在到底在做什么

游戏里的状态通过 Mod API 进入本地工作区。控制台把这些状态整理成玩家能看懂的页面，同时把玩家或 AI 的动作写入本地数据集。

基础 BC / 候选动作 AI 负责执行已经训练出来的策略。Phase 2A 微调会在出现 admitted self-play run 后增强候选动作模型。LLM 负责更像“大脑”的判断，但推荐模式下它不能自由构造动作；系统会先生成合法候选动作，LLM 只能从候选动作中选择。

现在最重要的不是无差别堆数据，而是稳定采集可复盘的高价值 Run，尤其是固定 seed 失败点、稀有卡关键回合、药水救命、路线/选卡分支和 admitted self-play run。

## 真实模块分工

| 模块 | 位置 | 作用 |
| --- | --- | --- |
| 游戏 Mod | `训练脚本/STS2MCP/` | 提供本地 API，读取场景、手牌、敌人、药水、奖励、地图，并写入 Run 数据。 |
| 控制台 | `AI_Training/control_panel.py` | 本地网页控制台，负责状态展示、进程启动、数据打包、训练触发。 |
| 基础 AI | `AI_Training/ai_agent.py` | 执行 BC 模型或规则给出的战斗/宏观动作。 |
| LLM Agent | `AI_Training/llm_agent.py` | 调用 OpenAI-compatible 接口，让模型在合法候选动作里选择。 |
| 数据管线 | `AI_Training/data_pipeline.py` | 把采集到的 jsonl 转成训练样本。 |
| Phase 2A 微调 | `AI_Training/train_rl_finetune.py` | 用 admitted self-play run 的评分权重微调候选动作模型。 |
| 模型包 | `AI_Training/ModelZoo/` | 存放可以提交到 GitHub 的演示模型包，控制台可切换。 |
| 原始数据 | `RL_Datasets/` | 本机采集的对局数据，默认不提交 Git。 |
| 数据包 | `Data_Packages/` | 一键打包数据库生成的 zip，默认不提交 Git。 |

## 控制台页面怎么读

| 区域 | 含义 |
| --- | --- |
| 顶部状态 | 先看“游戏连接”“AI 接管”“采集总开关”“当前 Run”。这里能判断游戏 API 是否在线、AI 是否会动、数据是否会写入。 |
| 实时采集动态 | 只显示最近几条动作，比如出牌、用药水、选卡、奖励、地图动作。它是观察窗口，不是完整日志。 |
| 战斗 AI | 控制基础 AI 进程和“允许 AI 出牌”。如果只想手动玩并采数据，保持关闭。 |
| LLM 模型接入 | 配置 Base URL、API Key、Model，控制是否启用模型决策，以及 LLM 是否真的执行战斗动作。 |
| 采集与训练 | 控制是否写入数据，选择最低训练质量，触发重建数据和重训。 |
| 自训练 | 固定 seed、目标局数、探索强度、入训批次和 run 评分。 |
| 右侧工作区 | 显示 AI 出牌逻辑、LLM 决策、模型切换、Run 数据体检、最近 Run、采集记录、评测和训练输出。 |

## 推荐演示流程

1. 双击 `start_all.bat` 或中文的一键启动脚本，打开控制台、日志窗口、BC AI 和 LLM。
2. 进游戏后先看顶部“游戏连接”。如果显示未连接，优先检查游戏、Mod 和本地 API。
3. 在“AI 模型状态”里确认 `demo_local_20260430` 完整可用；如果有多个模型包，可以在这里切换。
4. 如果只是采集人类数据：打开“采集总开关”，关闭“允许 AI 出牌”和“允许 LLM 自动战斗”。
5. 如果演示基础 AI：打开“允许 AI 出牌”，先不要打开宏观操作和商店购买。
6. 如果演示 LLM：先测试连接，再启用模型决策；动作选择用“只从合法候选动作里选”；确认建议靠谱后再打开自动战斗。
7. 一局结束后看“最近 Run”和质量标记。明显坏局要丢弃，正常局可以一键打包数据库提交给维护者。

## 关键开关说明

| 开关 | 说明 |
| --- | --- |
| 允许 AI 出牌 | 基础 AI 是否可以在战斗中自动出牌。这个开关不等于 LLM。 |
| 允许 AI 宏观操作 | 地图、奖励、选卡、事件、营火等战斗外行为。演示前期建议关闭。 |
| 采集总开关 | 是否把后续动作写入训练日志。调 UI 时可以关，正式采样要开。 |
| 自训练固定 Seed | 用来复现同一局面和死因；空 seed 则随机开局。 |
| 行为约束模式 | 自训练时控制规则护栏强度：保守、探索、自由。 |
| 启用模型决策 | 是否请求 LLM。关闭后 API 配置保留，但不会继续消耗请求。 |
| 动作选择 | 推荐“只从合法候选动作里选”。兼容模式保留给调试，不适合演示稳定性。 |
| 允许 LLM 自动战斗 | 打开后 LLM 建议会被执行；关闭时只显示建议和理由。 |

## 模型包切换

控制台的“AI 模型状态”里有“训练模型切换”。模型包位于：

```text
AI_Training/ModelZoo/<model_id>/
```

当前公开演示包：

| 模型包 | 内容 |
| --- | --- |
| `demo_local_20260430` | 战斗 BC 642 样本、候选动作 2948 行、宏观 BC 528 样本。 |

切换模型包会把包内文件复制到当前运行目录：

- `AI_Training/ProcessedParams/`
- `AI_Training/ProcessedMacroParams/`

切换后需要重启 AI，运行时才会重新加载模型。

## 数据和训练现在缺什么

现在最缺的是高质量 Run 和高价值 self-play run，尤其是一关 Boss 前后的不同牌组、不同药水、不同怪物组合、稀有卡关键回合和固定 seed 失败复盘。失败局也有价值，但必须标清楚质量，不能把卡死或误操作当成正常正样本。

Phase 2A 已经接入训练链路，但只有 `self_play_scores.json` 中 `admitted=true` 的 run 才会参与微调。当前 A1 死亡局用于复盘，不直接回灌模型。

怪物数据已经开始采集，后续会把怪物名称、意图、攻击模式和打法经验接入训练。这样同类怪物可以共享打法，而不是每次都只靠当前手牌做短视判断。

LLM 强化的重点不是让它自由发挥，而是让它读懂当前局面：手牌、弃牌堆、抽牌堆、药水、敌人意图、斩杀差、当前风险，然后在合法动作中选择更合理的一步。
